# MeshClaw Roadmap

Last updated: 2026-05-23

MeshClaw is an AI execution control plane. It is not the primary chat assistant
and it is not trying to replace Codex, Claude, Cursor, Open WebUI, SSH, Docker,
Kubernetes, or OpenClaw. Argos is the human-facing assistant identity that uses
MeshClaw Router/runtime for safe work.

The architecture map is `docs/ARCHITECTURE_2026-05-23.md`. Keep roadmap work
aligned with that layer/mode split.

The user talks to AI operators. MeshClaw gives those operators safe execution,
current state, permissions, policy decisions, use-only secrets, automation
adapters, and evidence.

```text
Codex / Claude / Cursor
  -> MeshClaw MCP / CLI
     -> policy, approvals, inventory, capabilities, workflows, evidence
     -> vssh / Guard / Argos UI Runner / browser / mail / calendar / APIs
        -> servers, devices, desktop apps, accounts, models, services

Signal / Argos / Open WebUI / local models
  -> MeshClaw Router / CLI
     -> policy, approvals, inventory, capabilities, workflows, evidence
     -> vssh / Guard / Argos UI Runner / browser / mail / calendar / APIs
        -> servers, devices, desktop apps, accounts, models, services
```

## Product Lines and Runtime Modes

MeshClaw Platform has three product lines that share a common core:

```text
MeshClaw Platform
├─ Argos Assistant    (personal assistant, briefings, documents, mail, calendar, Signal)
├─ MeshClaw Ops       (server, VM, and Docker operations automation)
└─ MeshClaw Agent     (lightweight agent installed on server nodes)
```

The shared core exposes runtime modes across the product lines:

| Mode | Product Line | Primary operator | Execution layer |
| --- | --- | --- | --- |
| Ops Mode | MeshClaw Ops | Codex, Claude, Cursor, Open WebUI | MeshClaw + vssh + MeshClaw Agent |
| Guard Mode | MeshClaw Platform (security feature) | local model, Signal Guard, human | MeshClaw Guard + OS keychain/pass/vault adapters |
| Automation Mode | Argos Assistant | Codex/Claude/local model/scheduler | MeshClaw + Argos UI Runner + domain adapters |

Names matter. MeshClaw Platform should not market itself as a personal
assistant. Argos Assistant is the user-facing assistant product line. MeshClaw
Ops is the server operations product line. MeshClaw Agent is the lightweight
node component. All three share the ability to let approved AI operators safely
execute real work.

## OpenClaw Boundary

OpenClaw is a personal AI assistant. MeshClaw is the runtime and safety layer
that AI assistants and coding agents can call.

OpenClaw-style risk appears when one agent owns conversation, memory, tools,
credentials, and execution. MeshClaw should deliberately split those concerns:

- conversation and reasoning: Codex, Claude, Cursor, Open WebUI, local models
- policy and approval: MeshClaw
- remote execution: vssh
- secret handling: Guard
- local desktop control: Argos UI Runner
- evidence and audit: MeshClaw

The central security rule:

```text
Chat context is not policy.
External content is not instruction.
MeshClaw policy is the execution boundary.
```

Prompt injection does not directly "attack MeshClaw" because MeshClaw is not a
language model. The real risk is an injected model calling MeshClaw tools. The
defense is to make high-risk tool calls require policy approval or be denied,
regardless of what the model says.

## Risk Classes

| Class | Default decision | Examples |
| --- | --- | --- |
| read-only | allow + evidence | status, facts, service audit, RSS read, calendar read |
| low-risk delivery | allow or approval-on-first-use | send redacted report, send voice file, call approved owner |
| account/content mutation | require approval | send email, move mail, create calendar event, edit document |
| infrastructure mutation | require approval or strong approval | restart service, quarantine service, rotate key, DNS change |
| destructive/cost | strong approval or deny | delete account, delete mail in bulk, rm -rf, reboot fleet, rent VPS |
| raw secret exposure | deny by default | reveal token/password to model, store raw secret in evidence |

Approvals must be structured records in MeshClaw evidence, not natural-language
statements buried in chat history.

## Controller Modes

MeshClaw must work in two practical operating modes.

### MacBook Controller Mode

```text
User -> Codex/Claude/Cursor on MacBook
     -> MeshClaw MCP
     -> Mac mini Argos UI Runner / fleet nodes
```

This is the strongest development and operations mode. The best model does the
reasoning. MeshClaw executes through stable adapters.

### Mac Mini Local Operator Mode

```text
Scheduler / Signal / local model on Mac mini
  -> MeshClaw CLI/MCP
  -> Argos UI Runner / local Ollama / fleet nodes
```

This mode handles unattended briefings, calls, monitoring, and routine reports.
Local models can summarize and classify, but high-risk actions still go through
MeshClaw policy.

## Argos UI Runner

Argos is the Signal identity on the Mac mini. Argos UI Runner is the stable
macOS app that receives one-time Accessibility/Screen Recording permissions and
performs local UI actions.

Rules:

- Keep a stable bundle id: `ai.meshclaw.argosrunner`.
- Do not run UI automation from temporary binaries or raw SSH AppleScript.
- Bind the runner to localhost by default.
- MeshClaw asks the runner for named actions, not arbitrary unreviewed tasks.
- The runner should fail closed when permission, target, screen state, or app
  state is uncertain.
- Every call attempt writes evidence.

First runner milestone:

1. health endpoint
2. request-accessibility endpoint
3. activate app endpoint
4. restricted click/key/type endpoints for development
5. Signal call state machine
6. browser/calendar/mail adapters using official APIs or Shortcuts before raw
   UI clicking

## Reusing Old Skills

Old MeshClaw "skills" can be reused, but not as free-form assistant skills.
Reclassify each old feature as one of:

- adapter
- workflow template
- capability definition
- policy rule
- evidence collector
- Guard action
- Argos UI action

Decision table:

| Old feature type | New home |
| --- | --- |
| server command/log/security task | Ops workflow or vssh adapter |
| mail/calendar/browser task | Automation workflow or Argos/domain adapter |
| password/token task | Guard workflow or vault adapter |
| notification/chat task | messenger delivery or approval channel |
| model prompt task | model capability, not MeshClaw brain |
| marketplace-style external code | archive until signed/sandboxed |

Keep, rewrite, archive, or delete decisions should be made with these questions:

1. Which mode owns it: Ops, Guard, or Automation?
2. Is it read-only, approval-required, strong-approval, or denied?
3. Does it produce structured JSON and evidence?
4. Does it expose raw secrets to models, logs, chat, or evidence?
5. Can Codex/Claude/OpenWebUI call it through MCP without parsing prose?

## Release Criteria

MeshClaw is product-ready for an early OSS release when:

- `meshclaw doctor` and `setup-check` are clear for a new user.
- MCP setup works in Codex, Claude, and Cursor without invalid tool names.
- `fleet-readonly-execute-demo --execute` produces useful evidence.
- `Argos UI Runner` can be installed and diagnosed cleanly on macOS.
- high-risk actions return approval-required, not accidental execution.
- Guard never reveals raw secrets by default.
- README explains the first 5 minutes.
- docs use English first and Korean can be added as separate guide text.
